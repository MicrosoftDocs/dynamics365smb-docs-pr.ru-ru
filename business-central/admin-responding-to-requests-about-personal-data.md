# <a name="responding-to-requests-about-personal-data"></a>Ответ на запросы о личных данных  
Субъекты данных могут запрашивать различные типы действий в отношении своих личных данных. Если вы классифицировали конфиденциальность своих данных и уверены, что они правильны, администратор может отвечать на запросы с помощью листа классификации данных в ролевом центре **Управление пользователями, группами пользователей и разрешениями** или, если вы используете клиент Windows, в ролевом центре, **Руководитель ИТ-отдела**. Дополнительную информацию о классификации данных и классификации конфиденциальности данных см. в разделах [Классификация данных](/dynamics-nav/classifying-data) и [Классификация чувствительности данных](admin-classifying-data-sensitivity.md).

В следующей таблице приведены примеры типов запросов, на которые можно отвечать.

> [!Note]
> Хотя мы обеспечиваем возможность отвечать на запросы этих типов и, таким образом, получать доступ к личным данным, ответственность за правильное размещение и классификацию личных и конфиденциальных данных лежит на вас.

|Тип запроса|Описание и предлагаемый отклик|
|-----|-----|
|Запросы переносимости|Субъект данных может выполнить запрос переносимости данных, означающий, частично, что вы должны экспортировать личные данные субъекта данных из вашей системы и предоставить их в структурированном обычно используемом формате. Для ответа на такие запросы используйте программу Data Privacy Utility для экспорта личных данных в файл Excel. С помощью Excel можно изменить личные данные и сохранить их в широко используемом формате, пригодном для компьютеров, например CSV или XML. Администраторы могут также экспортировать данные с использованием пакетов конфигурации Rapid Start, а затем настроить главные таблицы данных и связанные с ними таблицы, содержащие личные данные. |
|Запросы удаления|Субъект данных может запросить удаление своих личных данных. Существует несколько способов удалить личные данные с помощью настраиваемых возможностей, но вы отвечаете за принятие решения и реализацию. В некоторых случаях можно выбрать прямое редактирование своих данных, например, удаление контакта и затем выполнение пакетного задания удаления отмененных взаимодействий для удаления взаимодействий для этого контакта. <br><br> **Примечание.** Если указана дата в поле **Разрешить удаление документа до** на странице **Настройка модуля "Продажи"** или **Настройка модуля "Покупки"**, может потребоваться изменить дату, чтобы можно было удалить учтенные документы продаж и покупок, которые были напечатаны и которые имеют даты учета в эту дату или до нее.|
|Запросы на исправление|Субъект данных может запросить исправление ошибок в своих личных данных. Существует несколько способов сделать это. В некоторых случаях можно экспортировать списки в Excel для быстрого исправления нескольких записей, а затем импортировать обновленные данные. Дополнительные сведения см. в разделе [Экспорт бизнес-данных в Excel](https://docs.microsoft.com/en-us/dynamics-nav-app/about-export-data). Можно также вручную отредактировать поля, которые содержат личные данные, например изменить информацию о клиенте в карточке клиента. Однако записи транзакций, такие как общие операции, операции клиента и операции книги налогов, являются важными для целостности системы планирования ресурсов предприятия. Если личные данные хранятся в записях бизнес-транзакций, подумайте об использовании возможностей настройки для изменения таких личных данных.|

## <a name="restrict-data-processing-for-a-data-subject"></a>Ограничение обработки данных для субъекта данных
Субъект данных может запросить временную приостановку обработки своих личных данных. Для удовлетворения таких запросов можно пометить запись такого субъекта как заблокированную из-за конфиденциальности, чтобы остановить обработку их данных. Когда запись отмечена как заблокированная, вы не можете создавать новые транзакции, которые используют эту запись. Например, нельзя создать новый счет для клиента, если клиент или менеджер по продажам заблокирован. Чтобы пометить субъекта данных как заблокированного, откройте карточку субъекта данных, например карточку клиента, поставщика или контакта, и установите флажок **Заблокировано из-за конфиденциальности**. Может потребоваться выбрать **Показать больше** для отображения этого поля.

## <a name="handling-data-about-minors"></a>Обработка данных о несовершеннолетних
Если возраст контактного лица меньше официального возраста совершеннолетия в соответствии с законодательством вашего региона, это можно указать, установив флажок **Несовершеннолетний** на карточке **Контакт**. При этом автоматически устанавливается флажок **Заблокировано из-за конфиденциальности**. При получении согласия от родителей или официального опекуна несовершеннолетнего вы можете установить флажок **Родительское согласие получено**, чтобы разблокировать контакт. Хотя можно обрабатывать личные данные несовершеннолетних, нельзя использовать функцию профилирования в Microsoft Dynamics 365 for Sales.

> [!Note]
> Журнал изменений может регистрировать такие сведения, как когда и кем был установлен флажок **Родительское согласие получено**. Администратор может настроить это, используя руководство **Настройка журнала изменений**, и также выбрав флажок **Фиксировать в журнале изменение флажка полученного родительского согласия** в карточке **Контакт**. Дополнительные сведения см. в разделе [Регистрация изменений](/dynamics-nav-app/across-log-changes).
